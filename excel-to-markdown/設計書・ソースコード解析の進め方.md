# 設計書・ソースコード解析の進め方

「保守作業の改善検討テーマ」のテーマ1（影響範囲の把握・可視化）およびテーマ2（知識・仕様の継承・蓄積）を進めるにあたり、現状の設計書・ソースコードをどのように解析し、活用可能な形にするかをまとめる。

---

## 対象システムの構成

設計書は大きく2種類のシステムに対応している。

| 種別 | 実装技術 | 設計書の特徴 |
|---|---|---|
| **画面アプリケーション** | VB.NET | 画面項目・イベント・バリデーション等の記述が中心 |
| **バッチ処理** | PL/SQL | 処理フロー・テーブル操作・項目編集仕様の記述が中心 |

両者には共通する部分（テーブル定義への参照など）もあるが、設計書の構成・記述内容は大きく異なる。

### 着手順序

**バッチ処理（PL/SQL）の設計書から着手する。**

理由：
- バッチ処理の方が影響範囲の把握が難しく、解析の難易度が高い
- バッチで変換ルールを確立すれば、画面アプリケーションへの横展開が容易になる

---

## 基本方針

**設計書（Excel）をMarkdownに変換し、検索・横断参照が可能な状態にする。**

### なぜMarkdownか

| 観点 | Excel（現状） | Markdown（変換後） |
|---|---|---|
| 全文検索 | ファイルを1つずつ開いて検索 | grep / Obsidian検索で一括横断検索 |
| 横断参照 | 手作業で突合 | リンク・タグで機能間を相互参照可能 |
| 差分管理 | 取り消し線で履歴を残す（視認性低下） | Gitで差分管理、本文は常に最新 |
| 影響調査 | 関連箇所を記憶や目視で探す | キーワード検索で関連機能を即座に特定 |

---

## 現状の課題：取り消し線による履歴管理

現在の設計書は、修正履歴がわかるように**削除した文字列をそのまま残し、二重線（取り消し線）で消している**。

これにより以下の問題が起きている：

- 文書量が増え続け、**現在有効な記述と過去の記述の区別が読み取りにくい**
- 機械的な文字列抽出では、**削除済みの古い記述もヒットしてしまう**
- 新しく担当する人にとって、**どれが正しい記述かの判断に時間がかかる**

### 解決アプローチ

Excelのセル内テキストには書式情報（取り消し線の有無）がデータとして保持されている。
PowerShellからExcel COMオブジェクト（`Excel.Application`）を操作すれば、**取り消し線付きの文字列を判別して除外または分離**できる。

```
セルの文字列
├── 取り消し線なし → 現行の有効な記述 → Markdown本文として抽出
└── 取り消し線あり → 削除済みの記述  → 除外 or 別セクションに退避
```

これにより、**最新の有効な内容だけのMarkdownファイル**を生成でき、検索ノイズを排除できる。

## 進め方（3フェーズ）

### フェーズ1: 棚卸しと変換準備

**目的：** 何がどれだけあるかを把握し、変換の方針を決める

#### 1-1. 設計書の棚卸し

- バッチ処理の設計書ファイル一覧を作成する（ファイル名・対象機能・更新日）
- シートごとのレイアウトパターンを整理する
- 各シート・パターンについて「どのセル範囲から何の情報を取るか」を定義する

#### 1-2. 変換ツールのプロトタイプ作成

- PowerShell（またはVB.NET）で、代表的な設計書1〜2件を対象にMarkdown変換を試作する
- 取り消し線の判別ロジックを実装・検証する
- 変換結果の品質を目視確認し、ルールを調整する

#### 1-3. 着手範囲の決定

- 全設計書を一度に変換するのではなく、**影響調査で困る頻度が高い機能・領域**から着手する
- 優先度の判断基準：問い合わせ頻度、改修頻度、仕様不明度の高さ

### フェーズ2: 変換と構造化

**目的：** 設計書をMarkdownに変換し、検索・参照可能な形に整える

#### 2-1. Markdown変換の実施

##### バッチ処理設計書のシート構成と変換方針

| # | シート名 | 内容 | 変換方針 |
|---|---|---|---|
| 1 | **表紙** | 機能名等 | YAMLフロントマターのメタ情報として抽出 |
| 2 | **変更履歴**（改版履歴） | 版数・日付・変更内容 | 変換対象外（Excelの履歴管理はGit管理に移行） |
| 3 | **処理概要** | バッチの目的・処理フロー・入出力 | Markdownの本文として抽出（中核） |
| 4 | **処理仕様** | 処理の詳細ロジック・条件分岐・SQL | Markdownの本文として抽出（中核） |
| 5 | **項目編集仕様** | 出力項目の編集ルール・マッピング | 表形式 → Markdownテーブルに変換 |
| 6 | **補足説明** | 注意事項・例外・補足 | Markdownの本文として抽出 |
| - | *非表示シート* | - | **変換対象外** |

##### 変換時のポイント

- **表形式の内容**（項目編集仕様など）はMarkdownテーブル（`| col1 | col2 |`形式）に変換する
- **文章形式の内容**（処理概要・処理仕様など）はMarkdownの見出し・箇条書きに変換する
- **非表示シート**は `Sheet.Visible` プロパティで判定し、スキップする

#### 2-2. メタ情報の付与

各Markdownファイルに以下のメタ情報をYAMLフロントマターとして付与する：

```yaml
---
種別: バッチ処理設計書
機能名: ○○データ集計バッチ
実装: PL/SQL
関連テーブル:
  - T_ORDER
  - T_ORDER_DETAIL
  - T_SUMMARY
元ファイル: 設計書/バッチ設計_○○集計.xlsx
変換日: 2026-03-05
---
```

表紙シートから機能名等を自動抽出してフロントマターに設定する。
これにより、Obsidianのタグ・検索・Dataview等で横断的な参照が可能になる。

#### 2-3. 変換後のMarkdown出力イメージ

```markdown
---
種別: バッチ処理設計書
機能名: 受注データ集計バッチ
実装: PL/SQL
関連テーブル:
  - T_ORDER
  - T_ORDER_DETAIL
  - T_SUMMARY
元ファイル: 設計書/バッチ設計_受注集計.xlsx
変換日: 2026-03-05
---

### 1. 対象データ抽出

（処理仕様の詳細ロジックがここに続く）

## 項目編集仕様

| # | 出力項目 | 編集内容 | 入力元テーブル | 入力元カラム |
|---|---|---|---|---|
| 1 | 集計年月 | パラメータの基準日からYYYYMM形式に編集 | - | - |

```

#### 2-4. 取り消し線テキストの扱い

変換時に取り消し線テキストをどうするか、以下の選択肢から運用方針を決める：

| 方針 | メリット | デメリット |
|---|---|---|
| A. 完全に除外 | 本文がクリーンになる | 過去の経緯が追えなくなる |
| B. 折りたたみセクションに退避 | 経緯も参照可能 | 文書量はやや増える |
| C. 別ファイルに分離 | 本文はクリーン＋経緯も残る | ファイル数が倍になる |

**推奨：B案**（本文末尾に折りたたみで残す）

```markdown
## 処理概要
（現行の有効な記述のみ）

---

<details>
<summary>変更履歴（取り消し済みの記述）</summary>

- ~~旧：○○の場合は△△処理を行う~~
- ~~旧：□□テーブルを参照~~

</details>
```

### フェーズ3: 活用と運用

**目的：** 変換したMarkdownを実際の保守作業で活用し、継続的に育てる

#### 3-1. 影響範囲の調査への活用

- 修正対象のキーワード（テーブル名、関数名、画面ID等）でObsidian全文検索
- 関連する機能・バッチ・IFを即座に特定
- YAMLフロントマターの`関連テーブル`等で逆引き参照

#### 3-2. 知識の蓄積（テーマ2との連携）

Markdown変換した設計書に、保守作業を通じて得た知識を追記していく：

- 「この処理は○○の理由でこうなっている」（設計意図）
- 「ここを変更すると△△に影響がある」（影響メモ）
- 「□□の場合はこのSQLで確認可能」（調査ノウハウ）

設計書の「現行仕様」と、運用で得た「暗黙知」が同じ場所にまとまることで、テーマ2の目的も達成できる。

#### 3-3. 継続的な更新ルール

- 改修時：該当するMarkdownファイルも更新する（設計書とMarkdownの二重管理を防ぐルール）
- 新規機能：最初からMarkdownでも作成する（将来的にはMarkdownを正とする移行も検討）

## 変換ツール：技術的なポイント

### 使用技術の選択肢

| 技術 | 特徴 | 適したケース |
|---|---|---|
| **PowerShell + Excel COM** | 環境構築不要、試作が速い、スクリプトで手軽に実行 | プロトタイプ作成、少量〜中量の変換 |
| **VB.NET + Excel COM (Interop)** | コンパイルして配布可能、エラー処理が堅牢 | 本格的なツール化、大量の一括変換 |

**推奨：まずPowerShellで試作し、実用化の段階でVB.NETに移行する。**

PowerShellは即座に試して結果を確認できるため、変換ルールの検証・調整に向いている。
ルールが固まったらVB.NETでツール化すれば、他メンバーへの配布やバッチ実行も容易になる。

### Excel COMオブジェクトでの取り消し線判別

Excelのセル内テキストは、部分ごとに異なる書式を持つことがある（リッチテキスト）。
Excel COMオブジェクトでは、セル内の文字列を `Characters` プロパティで1文字ずつ、または範囲指定でフォント情報を取得できる。`Font.Strikethrough` プロパティで取り消し線の有無を判別する。

#### PowerShellでの実装イメージ

```powershell
$excel = New-Object -ComObject Excel.Application
$excel.Visible = $false
$workbook = $excel.Workbooks.Open("C:\path\to\設計書.xlsx")
$sheet = $workbook.Sheets.Item(1)

$cell = $sheet.Cells.Item(3, 2)  # B3セル
$totalLen = $cell.Characters().Count

$currentText = ""
$strikeText = ""

for ($i = 1; $i -le $totalLen; $i++) {
    $char = $cell.Characters($i, 1)
    if ($char.Font.Strikethrough) {
        $strikeText += $char.Text
    } else {
        $currentText += $char.Text
    }
}

# $currentText → 有効な記述（Markdown本文へ）
# $strikeText  → 取り消し済み記述（履歴セクションへ）

$workbook.Close($false)
$excel.Quit()
[System.Runtime.InteropServices.Marshal]::ReleaseComObject($excel) | Out-Null
```

#### VB.NETでの実装イメージ

```vb
Dim excelApp As New Microsoft.Office.Interop.Excel.Application()
excelApp.Visible = False
Dim workbook As Workbook = excelApp.Workbooks.Open("C:\path\to\設計書.xlsx")
Dim sheet As Worksheet = CType(workbook.Sheets(1), Worksheet)

Dim cell As Range = sheet.Cells(3, 2)  ' B3セル
Dim totalLen As Integer = cell.Characters.Count

Dim currentText As New StringBuilder()
Dim strikeText As New StringBuilder()

For i As Integer = 1 To totalLen
    Dim ch As Characters = cell.Characters(i, 1)
    If CBool(ch.Font.Strikethrough) Then
        strikeText.Append(ch.Text)
    Else
        currentText.Append(ch.Text)
    End If
Next

' currentText → 有効な記述（Markdown本文へ）
' strikeText  → 取り消し済み記述（履歴セクションへ）

workbook.Close(False)
excelApp.Quit()
Marshal.ReleaseComObject(excelApp)
```

### 取り消し線判別の注意点

- セル全体に取り消し線が設定されている場合：`Cell.Font.Strikethrough` が `True` で一括判定可能
- セル内の一部だけ取り消し線の場合：`Characters(start, length)` で部分的にフォントを確認する必要がある
- 結合セルの場合：結合範囲の左上セルに値が入っているため、結合情報（`MergeArea`）も考慮する

foreach ($sheet in $workbook.Sheets) {

### 表形式 → Markdownテーブル変換

項目編集仕様のような表形式のシートは、使用範囲（`UsedRange`）を走査してMarkdownテーブルに変換する。

```powershell
$range = $sheet.UsedRange
$rows = $range.Rows.Count
$cols = $range.Columns.Count

$mdTable = ""
for ($r = 1; $r -le $rows; $r++) {
    $line = "|"
    for ($c = 1; $c -le $cols; $c++) {
        $cell = $range.Cells.Item($r, $c)
        $text = Get-ActiveText $cell  # 取り消し線を除外した有効テキスト取得（自作関数）
        $line += " $text |"
    }
    $mdTable += $line + "`n"
    if ($r -eq 1) {
        # ヘッダー行の後にセパレータを挿入
        $sep = "|" + ("---|" * $cols)
        $mdTable += $sep + "`n"
    }
}
```

変換時の考慮事項：
- 結合セルは `MergeArea` で結合範囲を検出し、左上セルの値のみ取得する
- 空行が続く場合はテーブルの終端と判断する
- ヘッダー行の判定はシートのレイアウトパターンに応じて調整する

### 変換の自動化レベル

| レベル | 内容 | 対象 |
|---|---|---|
| 半自動 | PowerShellでテキスト抽出→手動で体裁調整 | レイアウトが不統一な設計書 |
| 自動 | レイアウトパターンに基づき完全自動変換 | パターン化された設計書 |

設計書のレイアウトがある程度パターン化されているとのことなので、**パターンごとに変換ルールを定義すれば高い自動化率が見込める**。

## 進め方：お試し変換 → 微調整の反復

評価は別環境で実施するため、以下のサイクルで段階的に精度を上げていく。

```
Round 1: 最もシンプルなバッチ設計書1件でお試し変換
           ↓  別環境で実行・結果確認
Round 2: 変換結果の問題点を洗い出し、スクリプトを修正して再変換
           ↓  別環境で実行・結果確認
Round 3: 少し複雑な設計書1〜2件で試す（例外パターンの発見）
           ↓  別環境で実行・結果確認
Round 4: 変換ルール確定 → 残りを一括変換
```

### お試し変換で選ぶ設計書

- **Round 1** は「自分がよく内容を知っている設計書」を選ぶ → 変換結果の正しさを素早く判断できる
- **Round 3** は「シートの使い方がやや特殊なもの」を選ぶ → 例外パターンの発見に有効

### お試し変換で確認する観点

| # | 確認観点 | 確認方法 |
|---|---|---|
| 1 | 取り消し線の判別 | 有効テキストと削除テキストが正しく分離されているか |
| 2 | 表形式の再現 | Markdownテーブルの列ずれ、結合セルの崩れがないか |
| 3 | 非表示シートのスキップ | 対象外のシートが出力に含まれていないか |
| 4 | 文字化け・改行 | セル内改行が意図通りに変換されているか |
| 5 | 検索性 | 変換後のMarkdownをObsidianで検索して、実際に使えるか |
| 6 | 可読性 | Markdownとして開いたときに読みやすいか |

### スクリプトの調整方針

スクリプトは**設定セクションとロジックセクションを分離**しており、調整時は主に設定セクションのパラメータを変更するだけで対応できるようにしている。詳細はスクリプトファイル（`ConvertExcelToMarkdown.ps1`）を参照。
